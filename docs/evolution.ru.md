## Legacy WMS приложение

### Предыстория

Данный проект был вдохновлён моим опытом работы .NET разработчиком в IT-компании, входившей в группу компаний одного крупного online-магазина цифровой техники. Я работал в отделе разработки WMS, делая приложения для внутренней логистики и складов. Одним из ключевых приложений было устаревшее Text-Based UI приложение, используемое для взаимодействия с терминалами сбора данных (ТСД).

### Мотивация автора и предпосылки создания

По мере того, как компания стремилась модернизироваться и внедрить современные CI/CD практики, возникли серьезные трудности с автоматизацией сборки и развертывания монолитных приложений на **.NET Framework 4.8**. Руководство начало исследовать возможности перевода основных проектов на актуальные версии платформы (на тот момент, **.NET 6/8**), и я решил взять старое Telnet UI приложение в свою зону ответственности.

Если честно, с самого начала это текстовое UI-приложение для склада показалось мне неудобным, специфичным и даже немного странным. Для любого разработчика, привыкшего к современному стеку, текстовый интерфейс (TUI) в 2020-х годах выглядит как архаичный динозавр:
* **Жесткие рамки:** Интерфейс зажат в строгую текстовую сетку (например, 24x16 символа).
* **Специфика ввода:** Полное отсутствие мыши — всё управление построено на стрелках, клавишах Enter/Esc, горячих кнопках и триггерах сканера штрихкодов.
* **Кажущаяся примитивность:** Сначала думаешь, что система представляет собой обычный вывод строк через `Console.WriteLine`, и исследовать здесь нечего.

Корни этого легаси-проекта уходили на 15 лет назад: изначально он вообще писался на **Visual Basic**, а затем был декомпилирован и кое-как переведен на C# (.NET Framework). Естественно, такой пласт истории нес в себе массу странных, спорных и неоднозначных решений.

Однако по мере поддержки проекта я начал погружаться во внутреннее устройство движка: механизмы отрисовки форм и виджетов, алгоритмы смены фокуса, навигацию, логику вызова делегатов для обработки событий и особенности рендеринга. В какой-то момент я поймал себя на мысли, что мне очень нравится внутренняя инженерия этого движка. Разумеется, речь не про бизнес-логику WMS (вряд ли кого-то могут до экстаза довести проводки паллет), а про изобретательность инженерных решений под капотом. Мне захотелось переосмыслить эти идеи на современном технологическом стеке.

### Визуальный пример формы

Внутри складского процесса форма на экране последовательно запрашивала данные у сотрудника, переключая фокус между виджетами ввода или отображая меню выбора. Типичный экран формы приемки товара выглядел примерно так:
```
+------------------------------------+
|             RECEIVING              |  <- TextWidget (header, usually inverted)
|                                    |
| PALLET: PO-10294                   |  <- TextWidget
| ITEM CODE: 2000192834017........   |  <- TextWidget + TextEntryWidget
| TARGET ZONE: A-10-02............   |  <- TextWidget + TextEntryWidget
|                                    |
| CHOOSE CONDITION:                  |  <- TextWidget (multiline)
| 1. GOOD CONDITION                  |
| 2. DAMAGED / DEFECT                |
| OPTION: 1..                        |  <- TextWidget + TextEntryWidget/ComboEntryWidget
|                                    |
| ITEM QUANTITY: .....               |  <- TextWidget + TextEntryWidget (focus is here)
|                                    |
|                                    |
| SCAN ITEM QUANTITY                 |  <- TextWidget (multiline hint for the active TextEntryWidget)
| ENTER - SKIP    ESC - BACK         |
+------------------------------------+
```

Хотя движок изначально проектировался под текстовый интерфейс, конечные пользователи (складские работники) редко взаимодействовали с системой напрямую через Telnet-клиенты:

* **Разработчики и тестировщики** использовали прямое Telnet-подключение через консоль для быстрой отладки, проверки бизнес-логики и ручного тестирования интерфейсов.
* **Линейный персонал (кладовщики)** работал с промышленными терминалами сбора данных (ТСД) брендов Zebra, Honeywell и др. На ТСД под управлением ОС Android было установлено кастомное мобильное приложение, написанное на **Xamarin**.

Это Xamarin-приложение подключалось к серверу, принимало текстовую разметку экрана и отрисовывало её на мобильном дисплее "один в один" с консольным представлением. При этом мобильный клиент обогащал текстовый интерфейс платформенными фичами, недоступными в стандартном Telnet:
1. **Звуковые уведомления:** Проигрывание разных звуковых сигналов при успешном сканировании штрихкода и при возникновении ошибки валидации.
2. **Цветовая индикация:** Подсветка критических ошибок или предупреждений яркими цветами для привлечения внимания.
3. **Отображение HTML страниц**: часто могло использоваться для страниц товаров.

### 1. Жизненный цикл форм и роль рефлексии в legacy-системе

В старом WMS-приложении не использовались IoC-контейнеры (Dependency Injection) или пулинг объектов. Логика работы с экранами строилась на базе динамического главного меню, завязанного на базу данных.

#### Как был устроен экран Главного Меню

На экране терминала сбора данных (ТСД) главное меню представляло собой простой нумерованный список доступных пользователю складских операций. Визуально в терминале это выглядело примерно так:
```
+------------------------------------+
|             MAIN MENU              |
|                                    |
| 1. RECEIVING                       |
| 2. PUTAWAY                         |
| 3. PICKING                         |
| 4. INVENTORY                       |
|                                    |
|                                    |
| 0. EXIT                            |
|                                    |
| SELECTED OPTION: ..                |
|                                    |
|                                    |
| CHOOSE OPTION     ESC - BACK       |
+------------------------------------+
```

#### Механизм работы меню и изоляция рефлексии

* **Управление через БД:** Список пунктов для этого экрана формировался динамически. Система делала запрос в БД, проверяла права сотрудника и выводила только разрешенные ему строки.
* **Использование рефлексии:** В таблице БД каждому пункту меню было сопоставлено текстовое наименование C#-класса соответствующей формы. Когда кладовщик выбирал пункт (например, нажимал `1`), система вычитывала имя класса из базы и создавала объект стартовой формы через рефлексию. Это позволяло системным аналитикам оперативно включать или выключать процессы прямо на уровне СУБД без пересборки приложения.
* **Переходы через оператор `new`:** Внутри самих складских процессов (когда стартовая форма уже открылась) рефлексия для навигации практически не применялась. При вводе данных в текстовые поля (например, в виджеты типа `TextEntryWidget`) новые формы не плодились по умолчанию — ввод обрабатывался обычным делегатом валидации в контексте текущего экрана. Новое окно создавалось только при необходимости переключить шаг процесса (например, открыть форму сканирования товара после формы выбора ячейки). В подавляющем большинстве случаев это делалось напрямую через оператор `new NextStepScreen()`.

### 2. Механика удержания состояния и сетевой Event Loop в legacy-системе

Архитектура старого фреймворка опиралась на постоянное (Stateful) TCP-соединение и посимвольную обработку ввода, что накладывало жесткие ограничения на масштабируемость.

#### Анатомия сетевого цикла (Event Loop)
Жизненный цикл сессии и отрисовки интерфейса на уровне фреймворка строился следующим образом:
1. **Авторизация подключения:** При подключении ТСД к серверу вызывался системный метод `AcceptSocket`. Внутри него инициализировалась сессия и создавался корневой объект формы. Ссылка на форму и её компоненты удерживалась непосредственно в методе, обслуживающем данный сокет.
2. **Посимвольное слушание:** Фреймворк вызывал блокирующий метод `Socket.Listen()`, который реагировал на **каждый введенный пользователем символ**.
3. **Цикл обработки (Event Loop):** Как только символ поступал от клиента, запускался внутренний цикл:
   * Фреймворк пробегался по дереву компонентов текущей формы;
   * Вызывались соответствующие делегаты обработки событий для виджетов;
   * Выполнялась отрисовка активного экрана в плоский массив символов;
   * Массив символов отправлялся обратно на ТСД по открытому TCP-каналу.
4. **Управление фокусом:** Виджет, находящийся в фокусе (например, `TextEntryWidget`), сохранялся в виде ссылки в поле самой формы (`FocusedWidget`). Фокус синхронизировался с клиентом (Telnet, Xamarin-приложением), после чего сервер снова засыпал на методе `Socket.Listen()`, ожидая следующий символ.

Посимвольный режим в легаси-системе использовался ради достижения "In-place" фокуса: символы печатались прямо внутри координат `TextEntryWidget`, что позволяло серверу на лету контролировать границы виджета (clipping) и прерывать ввод при превышении лимита длины. Также анализ каждого символа позволял перехватывать данные "на лету" и при желании делать автодополнение. Однако платой за это стал перезапуск полного цикла Event Loop и отправка кадра по сети на каждый нажатый символ.

#### Как удерживался Call Stack и граф объектов в памяти
Поскольку вся навигация и бизнес-логика строились на синхронных делегатах (например, валидация ввода открывала новую форму внутри себя), ссылки на дочерние экраны замыкались в цепочках этих делегатов.

Пока пользователь находился глубоко в меню, метод `AcceptSocket` продолжал удерживать ссылку на корневую форму, та — на свои виджеты, виджеты — на делегаты, а делегаты — на новые дочерние формы. Вся эта древовидная структура "жила" в памяти сервера ровно столько, сколько было активно TCP-соединение.

#### Последствия для масштабируемости (Цена Архитектуры)
Удержание полных графов объектов интерфейса для каждого пользователя привело к нерациональному и лавинообразному росту потребления RAM. Это создало критическое бутылочное горлышко:
* **Жесткий Rate Limiter:** Чтобы сервер не падал по нехватке ресурсов, на уровне конфигурации стояло жесткое ограничение — **не более 50 активных подключений** на один инстанс приложения. Для современных Highload-стандартов это микроскопическая цифра.
* **Сложное масштабирование:** Горизонтальное масштабирование системы под нужды крупного склада достигалось не за счет эффективного распределения нагрузки, а за счет экстенсивного развертывания огромного количества изолированных инстансов приложения, каждый из которых тратил гигабайты RAM на удержание всего 50 сессий.

## Первая (Stateful) версия PixelTerminalUI

*Примечание: `PixelTerminalUI` — это полностью независимый open-source проект, разработанный мной в свободное время. Он написан с нуля, не является копией коммерческих систем и не содержит кода или конфиденциальной информации моего предыдущего работодателя.*

Изначально проект задумывался как попытка воспроизвести существующее легаси приложение со всеми его нюансами, чтобы изнутри прочувствовать боль архитектуры и найти эффективное решение. Первая версия фреймворка во многом повторяла старые проблемы, но с одним важным отличием — транспортным слоем стал **REST API** вместо постоянного TCP/Telnet соединения.

Получился всё тот же SmartUI, где обработка событий и навигация строились на основе синхронных делегатов, жестко привязанных к жизненному циклу формы на сервере.

Однако взаимодействие разделилось на законченные "пакеты данных". Чтобы не терять состояние экранов между HTTP-запросами, мне пришлось реализовать управление сессиями на сервере через обычный `ConcurrentDictionary`, где ключом выступал UID сессии.
* **Проблема утилизации памяти:** Так как постоянного сокета больше не было, сервер не мог узнать, когда клиент закрыл приложение. Мне пришлось вручную писать логику TTL (Time-To-Live) и механизмы инвалидации сессий в словаре, чтобы избежать `OutOfMemory`.
* **Пакетная обработка ввода вместо посимвольного эхо-режима:** Перенос системы на HTTP/REST потребовал отказаться от постоянного удержания потока ввода сервером. В демонстрационном CLI-клиенте обработка была переведена на пакетный режим: ввод данных вынесен за пределы экрана — в отдельную строку под формой с префиксом `> `. Это позволило изолировать backend ядро от низкоуровневых задач клиентского UI (отслеживание каретки фокуса, локальный рендеринг символов внутри полей), однако снизило общую интерактивность интерфейса в рамках данного текстового прототипа.
* **Итог:** Этот вариант архитектуры "не взлетел". Попытка применить его на реальной работе показала, что для интеграции пришлось бы либо полностью переписывать сложнейшее legacy-ядро системы, либо заменять ядро на современное, но при этом вручную переписывать сотни существующих виджетов и форм. По трудозатратам это было одинаково долго и экономически нецелесообразно.

*Дисклеймер автора:* Безусловно, для промышленной Server-Driven UI (BDUI) системы ввод текста под формой является ограничением. В коммерческой реализации сервер обязан транслировать метаданные активного поля (абсолютные координаты `CursorX/Y` и `MaxLength`), а "тупой" клиент — осуществлять локальный эхо-ввод символов строго в границах виджета. Однако этот шаг намеренно был оставлен за рамками первой итерации: приоритет был отдан бэкенд-архитектуре, а не написанию сложного UI-движка для системной консоли.

### Как это работало (Концепция SmartUI)

В первой версии фреймворка вся логика разметки, управления фокусом и обработки событий оставалась на стороне сервера. Клиентское приложение (тонкий клиент) выполняло исключительно роль терминала отрисовки, отображая полученную от сервера готовую матрицу символов.

Разработчик описывал форму декларативно, а логику обработки ввода привязывал к синхронным делегатам виджетов (например, `EnterValidation`, наследуемый от базовой формы `TerminalScreen`).

Пример реализации стартового экрана в первой версии `PixelTerminalUI`:
```csharp
public class StartScreen : TerminalScreen
{
    private TextEntryWidget txtUserInput;

    protected override void InitializeComponent()
    {
        txtUserInput = new TextEntryWidget();
        txtUserInput.Name = nameof(txtUserInput);
        txtUserInput.Top = 14;
        txtUserInput.Left = 0;
        txtUserInput.EntireLine = true;
        txtUserInput.Hint = "PRESS ENTER TO CONTINUE";
        txtUserInput.EnterValidation = txtUserInput_EnterValidation;
        Widgets.Add(txtUserInput);
    }

    // Обработка ввода через синхронный метод-делегат
    private bool txtUserInput_EnterValidation()
    {
        switch (txtUserInput.Value)
        {
            case "-n":
                // Поток сервера блокировался здесь в ожидании новой формы,
                // удерживая весь Call Stack и контекст формы в оперативной памяти.
                ShowScreen(new frmLogin());
                break;
            case "-q":
                ShowInformation("Are you sure to exit?");
                break;
        }
        return true;
    }
}
```

#### Устройство базовой формы в SmartUI-архитектуре

Чтобы наглядно увидеть проблему смешения ответственности и удержания ресурсов в памяти, достаточно взглянуть на реализацию класса `TerminalScreen` из первой (Stateful) версии фреймворка. Этот класс сочетал в себе модель данных, логику рендеринга, ручное управление памятью и навигационный контекст:

```csharp
public abstract class TerminalScreen
{
    public string Name { get; set; } = "";
    public int Height { get; set; }
    public int Width { get; set; }
    public SessionInfo? SessionInfo { get; set; }
    
    // Прямая ссылка на родительский экран (удержание графа объектов)
    public TerminalScreen? ParentScreen { get; set; } 
    
    public List<TextWidget> Widgets { get; set; } = [];
    public TextEntryWidget? FocusedEntryWidget { get; set; }
    
    // Синхронные делегаты логики и переходов
    public Func<bool>? ShowValidation { get; set; }
    public Func<bool>? ScreenValidation { get; set; }
    public Action? ShowMainMenu { get; set; }
    public Action? ShowSettings { get; set; }
    public ScreenParameters? ScreenParameters { get; set; }

    public virtual void Show()
    {
        int count = Widgets.Count;

        // Попытка ручной оптимизации аллокаций из-за постоянного пересоздания форм
        TextWidget[] poolArray = ArrayPool<TextWidget>.Shared.Rent(count);
        try
        {
            if (!OnShowValidation()) return;

            SessionInfo!.CurrentScreen = this;
            SessionInfo.AssignEmptyDisplayedInfo();

            Widgets.CopyTo(poolArray, 0);
            Array.Sort(poolArray, 0, count, Comparer<TextWidget>.Create((x, y) =>
            {
                int res = x.Top.CompareTo(y.Top);
                return res != 0 ? res : x.Left.CompareTo(y.Left);
            }));
            
            ReadOnlySpan<TextWidget> sortedSpan = poolArray.AsSpan(0, count);
            FocusedEntryWidget ??= ConfigureWidgets(sortedSpan);

            ShowTextWidgets(sortedSpan);
            // ... логика отрисовки буфера
        }
        catch (Exception ex) { ShowError(ex.Message); }
        finally { ArrayPool<TextWidget>.Shared.Return(poolArray); }
    }

    // Тот самый метод, создающий бесконечный вложенный Call Stack
    public void ShowScreen(TerminalScreen screen)
    {
        try
        {
            SessionInfo.CurrentScreen = screen;
            screen.ScreenParameters = ScreenParameters ?? new ScreenParameters();
            screen.SessionInfo = SessionInfo;
            screen.ParentScreen = this; // Замыкание ссылки
            screen.Init();
            screen.Show(); // Поток проваливается во вложенный метод и блокируется здесь
        }
        catch (Exception ex) { ShowError(ex.Message); }
    }
    // ...
}
```

Аналогичным образом, виджеты также имели состояние, логику рендеринга и иногда ссылки на другие виджеты.

### Почему эта архитектура оказалась непригодной для распределенных систем?

Попытка перенести старую архитектурную модель "один в один" на рельсы REST API выявила три фундаментальные проблемы, которые делают проект нежизнеспособным в современных реалиях.

**1. Инфраструктурные и технические ограничения**:

* **Синхронность обработчиков и I/O-блокировки:** Методы рендеринга и обработчиков событий (такие как `EnterValidation`) оставались полностью синхронными. При необходимости выполнить сетевой запрос, обратиться к базе данных или совершить дисковую операцию, текущий поток сервера полностью блокировался. Это сводило на нет всю пропускную способность веб-сервиса под нагрузкой.
* **Невозможность горизонтального масштабирования:** Вся информация о сессии и граф открытых форм хранились исключительно в оперативной памяти конкретного инстанса приложения (`ConcurrentDictionary`). Если развернуть такой сервис в кластере за балансировщиком нагрузки, система ломалась: *Сервер 1* отрисовывал форму, а следующий запрос ввода от этого же пользователя улетал на *Сервер 2*. Поскольку *Сервер 2* ничего не знал об этой сессии, навигация прерывалась, и сессия падала.
* **Деградация оперативной памяти и Garbage Collector:** Удержание тяжелых графов объектов и цепочек вызванных форм в RAM между HTTP-запросами приводило к лавинообразному переполнению памяти при росте числа пользователей. Долгоживущие объекты принудительно переходили в старшие поколения кучи (Gen 1 / Gen 2), нагружая Garbage Collector и провоцируя долгие блокирующие паузы (`Stop-the-world`).

**2. Архитектурный тупик: Логический хаос и Спагетти-код**

Самым критичным фактором краха SmartUI-подхода стали не проблемы с производительностью, а **неуправляемая сложность кодовой базы**:

* **Сбой логики Event Loop при возврате экранов:** Наиболее неприятные и трудновоспроизводимые баги возникали в момент закрытия дочерней формы. Когда управление возвращалось в родительскую форму, фреймворк должен был восстановить фокус на исходном виджете. Однако из-за сохранения общего состояния в памяти внутренний Event Loop отрабатывал некорректно: значения полей не сбрасывались, повторно триггерились старые валидации, а фокус прыгал на случайные элементы.
* **Эффект "Черного ящика":** В результате stateful-подхода код превращался в монолитное спагетти, где логика отображения, управления фокусом, сетевого обмена и бизнес-валидации была намертво переплетена. Становилось абсолютно невозможно предсказать поведение формы при добавлении нового шага.

*Примечание: Аналогичное состояние кодовой базы наблюдалось и в оригинальном движке legacy WMS-приложения. Но там стабильность достигалась годами точечных доработок и непрерывного тестирования на реальном складе. Код работал, но превратился в священную корову, которую все боялись трогать. Повторять этот путь в новом open-source проекте не имело смысла.*

## Переход к Stateless: Разрыв Call Stack и Архитектура

Чтобы сделать систему масштабируемой и избавиться от спагетти-кода, я полностью отказался от удержания состояния в памяти сервера и перешел к проектированию Stateless архитектуры для движка. Она основывалась на трех принципах: внешнее хранение состояний (MongoDB), разделение данных и логики, а также использование изолированных рендереров. Больше никаких долгоживущих объектов форм, вложенных делегатов и заблокированных потоков.

Теперь каждый HTTP или gRPC запрос от клиента — это изолированная, короткая транзакция.

### 1. Декларативные интерфейсы: Рекорды данных
Интерфейс строится на C# рекордах (`record`), которые используются как пассивные структуры данных, сериализуемые в JSON. Примеры структур:
```csharp
public readonly record struct Pixel(
    char Symbol = ' ',
    bool IsInverted = false,
    ConsoleColor Foreground = ConsoleColor.White,
    ConsoleColor Background = ConsoleColor.Black);

public record TextWidget
{
    public required Guid Id { get; set; }
    public required string Name { get; set; }
    public required string Value { get; set; }
    public int Left { get; set; }
    public int Top { get; set; }
    public int Width { get; set; }
    public int Height { get; set; }
    public bool Visible { get; set; }
    public bool Inverted { get; set; }
    public virtual bool Editable { get; set; } = false;
    public int? TabIndex { get; set; }
    public ConsoleColor Foreground { get; set; } = ConsoleColor.White;
    public ConsoleColor Background { get; set; } = ConsoleColor.Black;
}

public record TextEntryWidget : TextWidget
{
    public override bool Editable { get; set; } = true;
    public bool Required { get; set; } = true;
    public char EmptyEnterSymbol { get; set; } = '.';
    public string? Hint { get; set; }
    public CommandBase? Command { get; set; }
}

public abstract record TerminalScreen
{
    public required Guid Id { get; set; }
    public required Guid SessionId { get; set; }
    public required string Name { get; set; }
    public bool Visible { get; set; } = true;
    public int Height { get; set; }
    public int Width { get; set; }
    public Guid? ParentScreenId { get; set; }
    public IEnumerable<TextWidget> Widgets { get; set; } = [];
    public Guid? FocusedEntryWidgetId { get; set; }
    public bool EnableDoubleBuffering { get; init; } = false;
}

public record SimpleMessageScreen : TerminalScreen;
```

#### Пример: Добавление кастомного поля без боли

Представим, что бизнес требует добавить на форму авторизации новое поле — индикатор уровня доступа (например, для отображения роли сотрудника). Всё, что требуется — расширить стандартный `TextWidget` путём обычного наследования и объявить новое свойство:
```csharp
public record ClassifiedTextWidget : TextWidget
{
    // Новое прикладное поле, необходимое для бизнес-логики безопасности
    public SecurityAccessLevel AccessLevel { get; set; } = SecurityAccessLevel.Standard;
}
```

После этого новый виджет сразу готов к использованию внутри любого экрана, например, в нашем стартовом `WelcomeScreen`, без изменения ядра движка:
```csharp
public sealed record WelcomeScreen : TerminalScreen
{
    public WelcomeScreen()
    {
        Name = "WelcomeScreen";
        Width = 40;
        Height = 10;
        
        Guid inputId = Guid.NewGuid();
        Widgets = new List<TextWidget>
        {
            new TextWidget { Left = 2, Top = 2, Value = "WELCOME TO THE GRID" },
            
            // Легко переиспользуем наш новый виджет с кастомным бизнес-свойством
            new ClassifiedTextWidget 
            { 
                Left = 2, Top = 3, 
                Value = "RESTRICTED AREA", 
                AccessLevel = SecurityAccessLevel.Critical,
                Foreground = ConsoleColor.Red

            new TextEntryWidget

            }
        };
        FocusedEntryWidgetId = inputId;
    }
}
```

### 2. Логика обработки: Команды (`ICommand`)

Бизнес-логика вынесена в атомарные `ICommand`, поддерживающие асинхронность и сериализуемый шаг выполнения. Реализована привязка команд к UI-элементам, позволяющая обрабатывать валидацию без необходимости хранения состояния в памяти сервиса.

Вот как это реализовано в ядре фреймворка:
```csharp
public interface ICommand
{
    Guid Id { get; }
    Guid WidgetId { get; set; }
    int RawState { get; set; }
    ValueTask<bool> ExecuteAsync(ICommandContext context);
}

public abstract class CommandBase : ICommand
{
    public abstract Guid Id { get; }
    public abstract Guid WidgetId { get; set; }
    public abstract int RawState { get; set; }
    public abstract ValueTask<bool> ExecuteAsync(ICommandContext context);
}

public abstract class Command<TEnum> : CommandBase where TEnum : struct, Enum
{
    public abstract TEnum State { get; set; }

    public override int RawState
    {
        [MethodImpl(MethodImplOptions.AggressiveInlining)]
        get
        {
            TEnum localState = State;
            return Unsafe.As<TEnum, int>(ref localState);
        }
        [MethodImpl(MethodImplOptions.AggressiveInlining)]
        set
        {
            int localValue = value;
            State = Unsafe.As<int, TEnum>(ref localValue);
        }
    }
}
```

Концепция реализации команд в `PixelTerminalUI` напрямую вдохновлена механизмами компилятора C# для генерации асинхронных методов (`IAsyncStateMachine`). При компиляции каждый `async`-метод преобразуется в стейт-машину:
*   Локальные переменные исходного метода становятся полями сгенерированной структуры/класса.
*   Добавляется целочисленное поле `_state` для отслеживания текущего шага выполнения.
*   Тело метода разделяется на блоки внутри конструкции `switch/case` в точках разрыва (`await`).

В новой архитектуре фреймворка этот же паттерн применен для обеспечения полной отказоустойчивости и распределенности бизнес-логики. Сама команда и её текущий шаг (`RawState`) представляют собой пассивные, сериализуемые данные. Они сохраняются во внешнее хранилище сессий вместе с текущим экраном.

Это позволяет в любой момент прервать выполнение процесса на одном узле кластера, сохранить промежуточную "точку останова" команды в БД, а при поступлении следующего сетевого пакета от клиента — восстановить команду и продолжить выполнение с того же шага на абсолютно другом сервере.

Для моделирования шагов выполнения используется строго типизированное перечисление (`enum`), задающее карту состояний команды:
```csharp
public enum MultiStepState
{
    Initial = 0,
    Processing = 1,
    Finalizing = 2
}
```

Ниже приведен пример реализации многошаговой команды `SequenceExecutionCommand`:
```csharp
public sealed class SequenceExecutionCommand : Command<MultiStepState>
{
    public override MultiStepState State { get; set; } = MultiStepState.Initial;
    public override Guid Id { get; } = Guid.NewGuid();
    public override Guid WidgetId { get; set; }

    public override ValueTask<bool> ExecuteAsync(ICommandContext context)
    {
        if (context == null)
        {
            return ValueTask.FromResult(false);
        }

        switch (State)
        {
            case MultiStepState.Initial:
                // --- ШАГ 1: Первая фаза валидации/обработки ввода ---
                // Имитация бизнес-валидации на начальном шаге обработки
                if (string.Equals(context.InputValue, "ABORT", StringComparison.OrdinalIgnoreCase))
                {
                    // Передаем кастомный текст ошибки бизнес-логики обратно в пайплайн
                    context.ErrorMessage = "Operational workflow sequence interrupted by user initial phase abort signal.";
                    return ValueTask.FromResult(false);
                }

                // Переводим команду на следующий шаг
                State = MultiStepState.Processing;
                
                // Модифицируем форму под ввод данных для следующего шага
                // [Логика обновления context.Screen]
                
                return ValueTask.FromResult(true);

            case MultiStepState.Processing:
                // --- ШАГ 2: Вторая фаза обработки ---
                // Имитация вторичной глубокой проверки в цикле транзакционной обработки
                if (context.InputValue.Length < 3)
                {
                    context.ErrorMessage = "The submitted processing execution verification key sequence token is too short.";
                    return ValueTask.FromResult(false);
                }

                // Переводим команду в финальную фазу
                State = MultiStepState.Finalizing;

                // Выполняем основное транзакционное действие
                // [Логика сохранения данных]

                return ValueTask.FromResult(true);

            case MultiStepState.Finalizing:
                // --- ШАГ 3: Завершение и очистка контекста навигации ---
                // [Логика возврата на предыдущую форму или в меню]
                
                return ValueTask.FromResult(true);

            default:
                return ValueTask.FromResult(false);
        }
    }
}
```

### 3. Паттерн перехвата ошибок (Short-Circuit Interception)

*Примечание: Центральным оркестратором всей системы является `RequestPipelineHandler`. Этот компонент отвечает за сквозной жизненный цикл обработки каждого действия пользователя: он принимает сырой ввод, координирует инфраструктурную валидацию, выполняет привязаные к виджетам бизнес-команды и в финале направляет мутировавший экран на Stateless-рендеринг для сборки сетевого ответа.*

Stateless-архитектура накладывает жесткие требования на обработку исключительных ситуаций. Если бизнес-команда или инфраструктурный валидатор завершаются ошибкой, мы не можем просто выбросить исключение (Exception) — это приведет к падению сетевого потока и потере сессии. Движок должен уметь «на лету» перехватывать сбой, подменять текущий экран на унифицированное окно уведомления и возвращать клиенту валидную матрицу пикселей.

#### Исключения (Exceptions) против Контролируемых Сбоев

Внимательный читатель спросит: *«А почему нельзя было просто обернуть весь `RequestPipelineHandler` в глобальный блок try-catch, перехватывать любые системные Exceptions и автоматически превращать их в экраны ошибок?»*

Это осознанное архитектурное ограничение фреймворка, разделяющее зоны ответственности:

1. **Разделение типов ошибок:** Инфраструктурные сбои (невалидный ввод) и бизнес-сбои (отсутствие товара) — это *штатные, прогнозируемые сценарии*. Они не должны генерировать тяжелые системные исключения. Прокидывание статуса через `bool` и контекст — это более высокопроизводительный путь, не нагружающий процессор сборкой стейктрейсов.
2. **Защита целостности данных:** Необработанное исключение (`Exception`) в прикладном коде разработчика — это аварийная ситуация (баг приложения, падение связи с СУБД, OOM). В этот момент состояние памяти и сессии становится недетерминированным. Если движок начнёт «гасить» такие исключения внутри себя и перерисовывать интерфейс, он замаскирует критический дефект. Аварийный запрос должен падать честно, чтобы сработали стандартные механизмы логирования хоста (Middleware/Logging), а транзакции в СУБД откатились.
3. **Ответственность прикладного слоя:** Обернуть потенциально опасный кусок кода (например, интеграционный запрос к внешнему API) в `try-catch`, обработать его и осознанно вернуть `false` с понятным текстом для пользователя — это прямая обязанность прикладного разработчика, пишущего конкретную команду. Фреймворк предоставляет для этого все инструменты (например, свойство `ErrorMessage` в контексте `CommandContext`).

#### Перехват на уровне инфраструктурной валидации

Первый рубеж обороны — это проверка входящих данных до того, как они попадут в бизнес-логику. Если инфраструктурный валидатор (например, проверка на максимальную длину строки или корректность символов) сообщает о сбое, движок мгновенно прерывает стандартный конвейер. Он вызывает фабрику ошибок и отправляет пакет на отрисовку:
```csharp
IEnumerable<ValidationDelegate> screenValidators = _validationProvider.GetValidatorsForScreen(screen.Name);
foreach (ValidationDelegate validate in screenValidators)
{
    ValidationResult validationResult = validate(screen, request.UserInput);
    if (!validationResult.IsValid)
    {
        // Перехватываем ошибку валидации и собираем форму через фабрику
        string errorMsg = validationResult.ErrorMessage ?? "Validation Fault!";
        SimpleMessageScreen errorScreen = _errorScreenFactory.BuildErrorScreen(sessionId, screen, errorMsg);
        
        await _sessionRepository.SaveActiveScreenAsync(sessionId, errorScreen);
        return await RenderAndBuildResponseAsync(errorScreen);
    }
}
```

#### Обработка сбоев в бизнес-командах (Паттерн Транспорта Ошибок)

Если валидация пройдена, управление передаётся прикладной команде. Бизнес-логика может завершиться неудачей (например, «Товар не найден на складе»). Чтобы передать текстовое описание ошибки наверх, используется свойство-транспорт `ErrorMessage` в локальном контексте команды:
```csharp
if (entryWidget.Command != null)
{
    CommandContext commandContext = new(
        sessionId: sessionId,
        screen: screen,
        focusedEntryWidget: entryWidget,
        inputValue: entryWidget.Value,
        sessionRepository: _sessionRepository
    );

    bool isExecutionSuccessful = await entryWidget.Command.ExecuteAsync(commandContext);
    if (!isExecutionSuccessful)
    {
        // Бизнес-логика вернула false. Сбрасываем невалидный ввод
        entryWidget.Value = string.Empty;
        await _sessionRepository.SaveActiveScreenAsync(sessionId, screen);

        // Извлекаем кастомную причину сбоя, записанную командой в контекст
        string businessError = commandContext.ErrorMessage ?? "Command Rejected Execution!";
        SimpleMessageScreen businessErrorScreen = _errorScreenFactory.BuildErrorScreen(sessionId, screen, businessError);
        
        await _sessionRepository.SaveActiveScreenAsync(sessionId, businessErrorScreen);
        return await RenderAndBuildResponseAsync(businessErrorScreen);
    }
}
```

Такое разделение изолирует логику формирования интерфейса от конвейера обработки. `RequestPipelineHandler` не знает, как выглядят виджеты ошибок и где они расположены на экране. Его задача — вовремя поймать сигнал сбоя (через `ValidationResult` или `commandContext.ErrorMessage`), переключить активный стейт на форму уведомления с помощью фабрики и гарантировать, что клиент получит корректно сформированный дельта- или full-пакет.

### 4. Сборка системы и декларативное расширение (DI / IoC)

Вся инфраструктурная сложность (управление рендерерами, побитовая упаковка, адаптивный расчет дельт) скрыта под капотом фреймворка. Для интеграции движка в хост-приложение (например, в стандартный Web API) прикладной разработчик использует декларативный Fluent API. 

Для этого предусмотрены три ключевых метода расширения для встроенного DI-контейнера `Microsoft.Extensions.DependencyInjection`:

#### 1. Инициализация ядра движка (`AddPixelTerminalUI`)
Базовый метод, который разворачивает в памяти Singleton-сервисы рендеринга и регистрирует встроенные компоненты «из коробки» (текстовые поля, поля ввода, маски паролей). На этом же этапе можно гибко управлять оптимизацией трафика:

```csharp
// Регистрация с дефолтными настройками (двойная буферизация включена)
builder.Services.AddPixelTerminalUI();

// Либо с осознанным отключением кэша кадров ради экономии CPU сервера
builder.Services.AddPixelTerminalUI(options =>
{
    options.EnableDoubleBuffering = false;
});
```

#### 2. Регистрация стартовой точки входа (`AddPixelTerminalStartup`)
Задает шаблон или конкретный тип экрана, который движок будет автоматически генерировать при холодном старте (когда пользователь только открыл терминал и сессия еще не создана).

```csharp
builder.Services.AddPixelTerminalStartup<WelcomeScreen>();
```
*Архитектурная деталь:* Под капотом фреймворк не использует рефлексию или тяжелые фабрики контекстов (`IServiceScopeFactory`). Вместо этого контейнер компилирует легковесный делегат активации `Func<Type, TerminalScreen>`, который порождает новые Transient-объекты экранов со скоростью нативного оператора `new`, полностью изолируя сессии пользователей друг от друга.

#### 3. Подключение сторонних плагинов и виджетов (`AddCustomTerminalRenderer`)
Движок спроектирован по принципу Open-Closed (открыт для расширения, закрыт для изменения). Если команде не хватает стандартных элементов, разработчик создает свой виджет, наследуясь от `TextWidget`, пишет для него изолированный рендерер и бесшовно внедряет его в общий конвейер одной строчкой:

```csharp
builder.Services.AddCustomTerminalRenderer<CustomWidget, CustomWidgetRenderer>();
```

При старте приложения движок автоматически соберет все зарегистрированные рендереры в потокобезопасный реестр. Когда `StatelessRenderer` встретит в дереве элементов ваш кастомный виджет, он мгновенно найдет нужную стратегию отрисовки, упакует данные в битовую матрицу и передаст их в адаптивный билдер ответов. Разработчик расширяет систему, вообще не прикасаясь к исходному коду ядра библиотеки.

Благодаря Fluent API, инициализация всего UI-движка, подключение распределенного хранилища состояний на базе MongoDB, а также явная регистрация форм и команд на бэкенде выглядят лаконично и декларативно:

Однако это ограничение является одной из ключевых причин, почему в целевой high-load архитектуре горячий стейт сессий и кадровые буферы двойной буферизации планируется полностью перенести в высокопроизводительное Key-Value хранилище Redis, где атомарный TTL-таймаут из коробки применяется ко всем связанным ключам.

### 5. Рендеринг: StatelessRenderer

За трансформацию дерева UI-компонентов формы в плоское представление отвечает глобальный синглтон `StatelessRenderer`. Вся логика отрисовки построена на концепции **одномерного линейного буфера пикселей**, аналогичной низкоуровневым паттернам работы с видеопамятью (Framebuffers).

Рендерер является полностью *Stateless*-сервисом: он не хранит внутри себя состояние экранов, не знает о существовании сетевых сессий и выполняет чистую математическую проекцию графа элементов на плоский сегмент непрерывной памяти за один проход, принимая целевой буфер извне.

Процесс формирования кадра устроен следующим образом:
* **Инициализация буфера:** Вызывающий слой выделяет плоский массив пикселей `Pixel[]` строго под суммарный объем сетки (`Width × Height`), а рендерер заполняет его дефолтными символами пробелов с пересчетом смещения строк по формуле `y * Width + x`.
* **Посимвольный Blitting (Перенос):** Фреймворк выполняет обход дерева виджетов текущей формы. Для каждого видимого элемента из IoC-контейнера извлекается специализированный процессор рендеринга (`IWidgetRenderer`), который рассчитывает абсолютные координаты и копирует символы с цветовыми атрибутами непосредственно в общий плоский массив.
* **Безопасность границ (Clipping):** Все операции проецирования защищены внутренними контурами проверки границ, что исключает падение движка по `IndexOutOfRangeException`, если координаты или ширина виджета выходят за физические размеры экрана ТСД.
* **Контекстные подсказки (Hint):** Если на форме есть виджет в фокусе, рендерер автоматически вычисляет индекс начала последней строки экрана, центрирует текст подсказки (`Hint`) и проецирует его на статус-бар.

### 6. Эволюция оптимизации рендеринга (Performance Roadmap)

#### Битовая упаковка и транспортный примитив (`uint`)

Передача двумерной символьной матрицы экрана в формате массива JSON-объектов на каждый пиксель накладывает критические ограничения на пропускную способность распределенной системы. Для стандартного терминального разрешения 80x24 кадровый буфер содержит 1920 логических пикселей. В текстовом формате JSON одна структура пикселя со всеми цветовыми атрибутами и флагами инверсии занимает около 85 байт (`1920 пикселей × 85 байт ≈ 163 КБ / кадр`):

```json
{
    "sessionId": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "screenBuffer": [
        {
            "symbol": " ",
            "isInverted": false,
            "foreground": 15,
            "background": 0
        },
        ...
    ],
    "width": 40,
    "height": 12
}
```

При интенсивном вводе данных пользователем (посимвольный сетевой обмен) или частом обновлении экранов на ТСД сетевой трафик моментально забивается мегабайтами избыточных данных, большая часть которых — это пустые символы пробелов со стандартным цветом фона.

Для минимизации сетевого оверхеда при сохранении полной гибкости визуализации (индивидуальный цвет и флаги для каждого пикселя) архитектура `PixelTerminalUI` переводится на паттерн битовой упаковки.

Вместо сериализации высокоуровневых объектов данные упаковываются в плоский бинарный массив примитивов `uint[]`. В .NET тип `char` занимает 16 бит. Используя быстрые побитовые сдвиги (`Bitwise operations`), оставшиеся 16 бит 4-байтового числа выделяются под метаданные цвета и системные флаги.

Схема упаковки структуры `UInt32` (32 бита):
```
Структура UInt32 (32 бита):
[ 16 бит: Символ (char) ] [ 7 бит: Foreground ] [ 7 бит: Background ] [ 2 бита: Флаги ]
```
При такой упаковке весь кадр размером 80x24 проецируется на фиксированный плоский массив примитивов, чистый объем которого в сети составляет: `1920 пикселей × 4 байта = 7.5 КБ`. Это обеспечивает **сокращение объема передаваемых данных почти в 22 раза** еще до применения алгоритмов сжатия HTTP-трафика (GZip/Brotli).

За упаковку и распаковку данных отвечает статический класс `PixelBitPacker` с принудительным инлайнингом методов компилятором (`AggressiveInlining`), что исключает накладные расходы на вызовы методов в рантайме:

```csharp
public static class PixelBitPacker
{
    private const uint CharMask = 0xFFFF;
    private const uint ColorMask = 0x7F;
    private const uint FlagsMask = 0x03;

    private const int CharShift = 16;
    private const int ForegroundShift = 9;
    private const int BackgroundShift = 2;

    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public static uint Pack(char character, byte foreground, byte background, byte flags)
    {
        return (uint)character << CharShift
               | (foreground & ColorMask) << ForegroundShift
               | (background & ColorMask) << BackgroundShift
               | flags & FlagsMask;
    }

    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public static void Unpack(uint packed, out char character, out byte foreground, out byte background, out byte flags)
    {
        character = (char)(packed >> CharShift & CharMask);
        foreground = (byte)(packed >> ForegroundShift & ColorMask);
        background = (byte)(packed >> BackgroundShift & ColorMask);
        flags = (byte)(packed & FlagsMask);
    }
}
```

В результате сериализованный JSON принимает максимально компактный вид, представляя собой простой массив числовых примитивов:

```json
{
    "sessionId": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "screenBuffer": [
        2104832,
        2104832,
        2104832,
        ...
    ],
    "width": 40,
    "height": 12
}
```

#### Оптимизация аллокаций памяти и пулинг буферов

Теоретический паттерн битовой упаковки пикселей требует аккуратного менеджмента памяти на уровне рантайма C#. Первоначальная концептуальная реализация движка (на базе двумерных массивов `Pixel[,]`) создавала потенциально высокую аллокационную нагрузку. При таком подходе на каждый такт сетевого Event Loop рендерер выделял в управляемой куче (Managed Heap) новый массив, который затем мапился в плоский `uint[]`. При гипотетическом росте нагрузок со стороны сотен или тысяч одновременно работающих терминалов постоянное выделение короткоживущих объектов приводило бы к избыточному давлению на Garbage Collector (GC) и блокирующим микропаузам рантайма.

Чтобы снизить аллокационный оверхед на этапе композиции кадра, конвейер рендеринга был перепроектирован:

* **Отказ от многомерных массивов:** В .NET массивы вида `Pixel[,]` имеют встроенный оверхед CLR на вычисление смещения индексов при каждом обращении, а также не поддерживаются стандартными пулами памяти. Движок был полностью переведён на работу с плоским одномерным массивом `Pixel[]`.
* **Симметричный пулинг через `ArrayPool<T>`:** Жизненный цикл промежуточного холста теперь изолирован внутри вызывающего метода по правилу «где арендовали, там и вернули». Сам рендерер (`StatelessRenderer`) больше ничего не выделяет в куче, а выполняет посимвольный перенос (blitting) прямо во внешний предоставленный массив. После упаковки данных в `uint[]` арендованный массив `Pixel[]` мгновенно возвращается в `ArrayPool<Pixel>.Shared.Return()`, минимизируя время его удержания в памяти.

##### Предыдущий вариант реализации (аллокация `Pixel[,]` внутри рендерера):
```csharp
private TerminalResponse RenderAndBuildResponse(TerminalScreen screen)
{
    Pixel[,] buffer = renderer.Draw(screen);

    int width = buffer.GetLength(0);
    int height = buffer.GetLength(1);

    uint[] flatBuffer = new uint[width * height];
    for (int y = 0; y < height; y++)
    {
        for (int x = 0; x < width; x++)
        {
            Pixel currentPixel = buffer[x, y];
            byte inversionFlag = (byte)(currentPixel.IsInverted ? 1 : 0);

            // Прямая упаковка пикселей в непрерывный массив примитивов.
            flatBuffer[y * width + x] = PixelBitPacker.Pack(
                currentPixel.Symbol,
                (byte)currentPixel.Foreground,
                (byte)currentPixel.Background,
                inversionFlag);
        }
    }
    return new TerminalResponse(screen.SessionId, flatBuffer, width, height);
}
```

##### Оптимизированный вариант реализации (с плоским буфером и `ArrayPool`):
```csharp
private TerminalResponse RenderAndBuildResponse(TerminalScreen screen)
{
    int width = screen.Width;
    int height = screen.Height;
    int totalCellsCount = width * height;

    Pixel[] pooledBuffer = ArrayPool<Pixel>.Shared.Rent(totalCellsCount);
    uint[] flatBuffer = new uint[totalCellsCount];
    try
    {
        renderer.Draw(screen, pooledBuffer);

        for (int index = 0; index < totalCellsCount; index++)
        {
            Pixel currentPixel = pooledBuffer[index];
            byte inversionFlag = (byte)(currentPixel.IsInverted ? 1 : 0);

            // Сжатие в целевой массив транспортного уровня.
            flatBuffer[index] = PixelBitPacker.Pack(
                currentPixel.Symbol,
                (byte)currentPixel.Foreground,
                (byte)currentPixel.Background,
                inversionFlag);
        }
    }
    finally
    {
        ArrayPool<Pixel>.Shared.Return(pooledBuffer);
    }

    return new TerminalResponse(screen.SessionId, flatBuffer, width, height);
}
```

#### Изоляция транспортного уровня

Важная архитектурная особенность обновлённого движка — полное отделение ядра рендеринга и обработки команд от сетевого (транспортного) слоя. На уровне фреймворка входной точкой является абстрактный `TerminalRequest`, а выходной — полиморфный `TerminalResponse`. 

Благодаря такой изоляции, поверх существующей системы можно развернуть практически любой протокол передачи данных:
* **HTTP (REST/RPC):** Классический stateless-обмен, идеально подходящий для простых сценариев «запрос-ответ».
* **WebSockets / SignalR:** Позволяет организовать постоянное полнодуплексное соединение для мгновенной доставки кадров.
* **gRPC (Streaming):** Оптимальный выбор для высоконагруженных промышленных сред, снижающий накладные расходы на сетевые заголовки.

Сам движок остаётся "чистой функцией", а выбор транспорта становится исключительно вопросом конфигурации хоста (инфраструктурной обвязки).

#### Механизм гибридного адаптивного рендеринга (Double Buffering)

Для минимизации сетевого трафика в архитектуру движка заложен механизм двойной буферизации. Фреймворк удерживает в персистентном хранилище снимок предыдущего отправленного кадра сессии. При обработке пользовательского ввода сервер вычисляет разницу (`diff`) между текущей и исторической матрицами. Идея проста: вместо перерисовки всего экрана целиком, передать клиенту только изменившиеся пиксели и их координаты (дельта-апдейт).

```json
{
  "$type": "delta",
  "mutations": [
    {
      "index": 162,
      "packedValue": 7675392
    },
    {
      "index": 163,
      "packedValue": 7544320
    },
    ...
  ],
  "sessionId": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "width": 40,
  "height": 12
}
```

На практике текстовый формат JSON накладывает жесткие ограничения на точечные обновления. Чтобы отправить один изменившийся пиксель, нам нужно передать не только его новое значение, но и служебные метаданные: название полей, кавычки, фигурные скобки и индекс ячейки. Из-за этого текстовая дельта начинает стремительно "раздуваться".

Давайте наглядно посчитаем эту математику на примере небольшого экрана размером 40x12 (всего 480 ячеек):

* **Вариант А: Передаем весь экран целиком (`FullFrameResponse`)**  
  В JSON это выглядит как обычный плоский массив чисел: `[12345,67890,11223,...]`. Одно значение с учетом разделительной запятой весит в среднем 8 байт.  
  *Итоговый размер кадра:* 480 ячеек × 8 байт = **3.8 КБ**. Этот вес фиксирован и не зависит от того, что происходит на экране.

* **Вариант Б: Передаем только точечные изменения (`DeltaResponse`)**  
  Каждое изменение сериализуется как массив объектов с жесткой структурой: `[{"Index":12,"PackedValue":12345},...]`. Из-за сильного оверхеда JSON-разметки (ключи, двоеточия, скобки) всего одна такая запись весит в среднем **36 байт**.  

А теперь сопоставим цифры: если мы разделим общий вес экрана (3800 байт) на вес одной мутации (36 байт), то получим критическую точку в **105 ячеек**. Таким образом, если на экране меняется всего **22% пикселей**, точечный JSON-пакет дельты начинает весить **больше**, чем отправка всей матрицы целиком.

Однако в реальности трафик сжимается на уровне сети (GZIP / Brotli). Как это меняет картину?

- Когда мы отправляем дельту, служебные ключи `"index"` и `"packedValue"` схлопываются практически в ноль алгоритмами сжатия благодаря поиску повторяющихся строк. Но стоит понимать, что в дельте остаются уникальные значения самих индексов (например, `162`, `163`), которые сжимаются плохо. 
- В то же время, полный кадр терминала (`screenBuffer`) зачастую забит огромными цепочками одинаковых чисел — пустых ячеек пробелов на черном фоне. Сетевые архиваторы сжимают такие непрерывные последовательности дубликатов с колоссальной эффективностью. 

С учетом транспортного сжатия реальная экономическая выгода точечной дельты окончательно теряется в районе **25% изменений**. Именно этот прагматичный порог в `0.25` и зафиксирован в компоненте, который отвечает за отрисовку ответа. Если изменений мало — летит компактная дельта, если экран перерисовывается масштабно (открытие новой формы) — движок мгновенно сбрасывает оптимизацию и шлет плоский полный кадр, который в сжатом виде весит меньше раздутой уникальными индексами дельты.

##### Обратная сторона двойной буферизации: Нагрузка на вычислительные ресурсы

Двойная буферизация даёт колоссальные преимущества для стабильности и плотности сетевого трафика, но переносит накладные расходы на сторону самого сервиса. При рендеринге "as is" (без буферизации) движок работает линейно: аллоцирует один массив кадра, заполняет его и сразу сериализует. 

Внедрение Double Buffering драматически усложняет конвейер на сервере:
1. Из базы данных/кэша вычитывается массив предыдущего кадра.
2. Аллоцируется и заполняется массив текущего кадра.
3. Происходит полный проход по всем элементам матрицы для вычисления диффа (вычислительная сложность `O(N)`).
4. Выделяется отдельный массив под мутации (`PixelMutation[]`).

В условиях жестких ограничений по CPU на сервере или при использовании сверхбыстрых локальных сетей (где трафик не является бутылочным горлышком) этот механизм может оказаться невыгодным.

Для гибкого управления этой оптимизацией в движок был заложен Fluent IoC-конфигуратор, позволяющий полностью включать/отключать двойную буферизацию на уровне инфраструктурной сборки приложения.
```csharp
// Инициализация ядра PixelTerminalUI с явным управлением двойной буферизацией
builder.Services.AddPixelTerminalUI(options =>
{
    options.EnableDoubleBuffering = true; // Управление оптимизацией трафика (true/false)
});
```

### Ограничения, компромиссы и цена решений (Trade-offs)

#### Смещение нагрузки на СУБД
Переход к архитектуре Stateless полностью снял проблему утилизации RAM на серверах приложений, но перенёс эту нагрузку на слой хранения. Теперь на каждый пользовательский ввод (клик, скан) движок вынужден выполнять связку операций чтение/запись в БД.

На текущих масштабах системы MongoDB справляется без проблем благодаря точечным индексам и микроскопическому размеру стейта (несколько килобайт на сессию). Однако при росте нагрузки в десятки раз база данных неизбежно станет бутылочным горлышком. Это осознанный компромисс: в продакшн-системах горячий стейт и кадровые буферы двойной буферизации должны сразу выноситься в ультрабыстрые Key-Value хранилища вроде Redis или KeyDB, оставляя MongoDB роль архивного репозитория.

#### Исторический контекст: почему не WebSockets + HTML/JS
Данный проект — это попытка переосмыслить реальный бэкграунд крупных legacy-WMS систем, зародившихся 15–20 лет назад. В те годы не было ни современных веб-стеков, ни мощных процессоров внутри ТСД (терминалов сбора данных). Железо было слабым, а складской Wi-Fi между экранирующими стеллажами постоянно обрывался. 

В таких условиях самописный Server-Driven UI (SDUI), возвращающий клиенту готовые матрицы пикселей, был единственным способом заставить систему работать. Клиентское приложение на ТСД оставалось "глупым" и максимально лёгким, оно не тратило заряд батареи на парсинг тяжелого HTML/JS и логику UI, а сервер брал всю работу (фокус, валидацию, отрисовку виджетов) на себя. При обрыве связи "глупый" клиент мгновенно восстанавливал картинку с того же места. 

В современной разработке новых систем с нуля глупо использовать TUI-матрицы при наличии Flutter, WebSockets или gRPC. Но как инструмент эволюции и спасения уже существующего легаси-кода — этот подход жизнеспособен.

#### "Велосипедостроение" (Битовая упаковка и свой протокол)
Побитовое сжатие цвета, символов и флагов в один `uint` primitive, как и разработка гибридного адаптивного билдера ответов — это попытка выжать максимум производительности в условиях изначально несовершенной архитектуры текстового JSON. 

Вместо того чтобы полностью менять формат обмена данными на бинарный (например, Protocol Buffers), система пытается балансировать внутри текстового протокола. Это классический компромисс между читаемостью (JSON легко отлаживать в веб-интерфейсах) и плотностью трафика на сетевом уровне.

#### Применимость в реальном бизнесе
Давайте смотреть на вещи прагматично: внедрить подобный движок в работающий крупный склад или производство "в лоб" практически невозможно. Существующие бизнес-процессы и кодовая база легаси-систем настолько огромны и переплетены, что их переписывание под Stateless-рельсы потребует колоссальных бюджетов и сопряжено с огромными рисками остановки отгрузок. 

Поэтому `PixelTerminalUI` стоит рассматривать не как готовый продукт для немедленной миграции вашего продакшена, а как концептуальное инженерное исследование (PoC — Proof of Concept). Проект доказывает, что даже старые, казалось бы, тупиковые Server-Driven UI архитектуры можно существенно оптимизировать по памяти и трафику, применив современные паттерны платформы .NET (`ArrayPool`, `Span<T>`, битовые маски и гибридную доставку кадров).
